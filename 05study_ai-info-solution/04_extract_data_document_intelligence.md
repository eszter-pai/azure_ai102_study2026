## 1. What Is Azure Document Intelligence?

> **Azure Document Intelligence** = a Foundry Tools cloud AI service that uses **OCR + deep learning** to extract text, key-value pairs, **selection marks**, and **tables** from documents, returning structured JSON output.

| Technology | Role |
|---|---|
| **OCR** | Creates **bounding boxes** around detected objects; records coordinates relative to the page |
| **Deep learning models** | Pre-trained on thousands of form examples, enables high-accuracy extraction with **little/no custom training** |

### Three Service Components

| Component | Description |
|---|---|
| **Document analysis models** | Extract text, tables, selection marks, structure; inputs: JPEG, PNG, PDF, TIFF; output: JSON |
| **Prebuilt models** | Pre-trained on specific form types (W-2, invoices, receipts, ID docs, business cards) |
| **Custom models** | Trained on your own business-specific forms via Document Intelligence Studio |

### Access Methods

| Method | Notes |
|---|---|
| REST API | Any language that supports HTTP |
| Client library SDKs | Python, C#/.NET, Java, JavaScript |
| Azure Document Intelligence Studio | Visual web UI for exploring, labeling, training, and testing |
| Microsoft Foundry | Integrated in the Foundry portal |

## 2. Resource Provisioning

> Two resource options: choose based on whether you need multiple AI services or just Document Intelligence.

| Resource Type | Use When |
|---|---|
| **Foundry Tools resource** | Access multiple Foundry Tools under a single endpoint and key |
| **Azure Document Intelligence resource** | Single-service access only; also required if using **Microsoft Entra authentication** |

> If you need **Microsoft Entra authentication**, you must use a **single-service** (Azure Document Intelligence) resource

## 3. File Input Requirements

| Requirement | Value |
|---|---|
| **Supported formats** | JPG, PNG, BMP, PDF (text or scanned), TIFF |
| **Max file size** | 500 MB (paid S0 tier) / 4 MB (free F0 tier) |
| **Image dimensions** | 50 × 50 px minimum → 10,000 × 10,000 px maximum |
| **Training dataset size** | 500 pages or less total |

## 4. Choosing the Right Component

| Use Case | Recommended Component |
|---|---|
| OCR: extract text, structure, tables, layout | **Document analysis model** (Read, Layout, or General Document) |
| Common form types (W-2, invoice, receipt, ID, health insurance, business card) | **Prebuilt model** |
| Industry-specific or custom business forms | **Custom model** |

## 5. Training a Custom Model

> Two ways to train: via the **REST API/SDK** (manual setup) or via **Document Intelligence Studio** (recommended).

### Option A: API/SDK Approach: 4 Steps

| Step | Action |
|---|---|
| **1** | Store sample forms in an **Azure Blob container** alongside JSON label files |
| **2** | Generate a **SAS URL** for the container |
| **3** | Call the **Build model** REST API (or SDK `begin_build_model()`) |
| **4** | Call **Get model** to retrieve the trained **Model ID** |

### Required Training Files (generated per sample form)

| File | Description |
|---|---|
| `ocr.json` | One per sample form: layout/OCR data; generated by **Analyze document** function |
| `fields.json` | One total: lists all fields you want to extract |
| `labels.json` | One per sample form: maps fields to their location in that form |

> In **Document Intelligence Studio**, these files are **created and stored automatically**.

### Option B: Document Intelligence Studio (Recommended)

Steps:
1. Upload at least **5–6 sample forms** to your Azure Blob storage container
2. Configure **CORS**(Cross-Origin Resource Sharing. It's a browser security rule that blocks web pages from making requests to a different domain than the one they loaded from.) on the **container** (allows Studio to write label files)
3. Create a **custom model project** in Studio: link your storage container + resource
4. **Apply labels** to text in Studio
5. **Train** the model → receive **Model ID** + **Average Accuracy per tag**
6. **Test** on a new form not used in training

## 6. Custom Model Types

| Model Type | Best For | Key Traits |
|---|---|---|
| **Custom template model** | Forms with consistent, predictable layout | Fast training (minutes); 100+ languages; extracts key-value pairs, tables, selections, signatures |
| **Custom neural model** | Semi-structured or **unstructured** documents | **Deep learning**; combines layout + language features; handles variable layouts |

## 7. Using Models via Python SDK

> Calls are **asynchronous**: begin analysis → wait for result.

```python
from azure.ai.formrecognizer import DocumentAnalysisClient
from azure.core.credentials import AzureKeyCredential

endpoint = "YOUR_DOC_INTELLIGENCE_ENDPOINT"
key = "YOUR_DOC_INTELLIGENCE_KEY"
model_id = "YOUR_CUSTOM_BUILT_MODEL_ID"
form_url = "YOUR_DOCUMENT_URL"

client = DocumentAnalysisClient(
    endpoint=endpoint,
    credential=AzureKeyCredential(key)
)

# Submit document for analysis (async)
task = client.begin_analyze_document_from_url(model_id, form_url)
result = task.result()
```

| Parameter | Description |
|---|---|
| `model_id` | Use a custom model ID, or a prebuilt model string (e.g., `"prebuilt-invoice"`) |
| `form_url` | Publicly accessible URL of the document |
| `task.result()` | Blocks until analysis completes; returns `AnalyzeResult` |

## 8. JSON Response Structure

> A successful response contains an `analyzeResult` object with extracted content and page data.

```json
{
  "status": "succeeded",
  "createdDateTime": "...",
  "lastUpdatedDateTime": "...",
  "analyzeResult": {
    "apiVersion": "2022-08-31",
    "modelId": "DocIntelModel",
    "content": "Purchase Order\nHero Limited\n...",
    "pages": [
      {
        "pageNumber": 1,
        "angle": 0,
        "width": 1159,
        "height": 1486,
        "unit": "pixel",
        "words": [
          {
            "content": "Purchase",
            "polygon": [89, 90, 174, 91, 174, 112, 88, 112],
            "confidence": 0.996,
            "span": { "offset": 0, "length": 8 }
          }
        ]
      }
    ]
  }
}
```

### Key Response Fields

| Field | Description |
|---|---|
| `status` | `"succeeded"` / `"failed"` |
| `analyzeResult.modelId` | ID of the model that analyzed the document |
| `analyzeResult.content` | Full extracted text as a string |
| `analyzeResult.pages[]` | Per-page data |
| `pages[].words[]` | Individual words with text, bounding polygon, confidence, span |
| `pages[].angle` | Rotation angle of the page |
| `word.polygon` | Bounding box as array of x,y coordinates |
| `word.confidence` | Confidence score (0–1) |

---

## 9. Understanding Confidence Scores

| Confidence Level | Guidance |
|---|---|
| **Low** | Input document quality may be poor; form appearance may differ from training data |
| **~80%+** | Acceptable for low-risk applications |
| **100%** | Recommended for sensitive cases (medical records, billing statements) |

> If confidence is low and the form layout varies, consider training **multiple models** — one per form format.

## 10. Azure Document Intelligence Studio

> A visual web interface for the complete workflow: explore → analyze → label → train → test.

### Studio Capabilities

| Category | Projects Available |
|---|---|
| **Document analysis models** | Read, Layout, General Documents |
| **Prebuilt models** | W-2, Invoice, Receipt, ID, Health insurance, Vaccination, Business card |
| **Custom models** | Build, label, train, and test custom models |

### Workflow by Project Type

| Project Type | Steps |
|---|---|
| **Document analysis** | Create resource → Select model (Read/Layout/General) → Provide endpoint + key → Analyze |
| **Prebuilt model** | Create resource → Select prebuilt type → Provide endpoint + key → Analyze |
| **Custom model** | Create resource → Upload 5–6 samples → Configure CORS → Create project → Label → Train → Test |

> In Studio, **ocr.json**, **labels.json**, and **fields.json** files are **automatically created** and stored in your storage account — no manual JSON file creation needed.

## 11. Document Intelligence vs Content Understanding (Comparison)

| Aspect | Azure Document Intelligence | Azure Content Understanding |
|---|---|---|
| **Service type** | Foundry Tools (OCR + deep learning) | Generative AI (schema-based) |
| **Content types** | Documents / forms | Documents, **images, audio, video** |
| **Prebuilt models** | Many (invoice, receipt, ID, W-2, etc.) | Prebuilt analyzer templates |
| **Custom training** | Custom template + neural models | Schema definition + minimal samples |
| **Training data needed** | 5+ labeled sample forms | Often 1 sample (generative AI) |
| **Studio** | Document Intelligence Studio | Content Understanding Studio |
| **Output** | JSON with bounding boxes + fields | JSON with schema-defined fields |

## Quick Reference — Exam Tips

| Key Fact | Detail |
|---|---|
| **3 service components** | Document analysis models, Prebuilt models, Custom models |
| **Microsoft Entra auth requirement** | Requires a **single-service** Azure Document Intelligence resource (not Foundry Tools) |

| **CORS** | Must be configured on blob container for Studio to write label files |
| **Custom template model** | Best for consistent/structured layouts; fast training; 100+ languages |
| **Custom neural model** | Best for semi-structured or unstructured documents |
| **Confidence threshold** | 80% for low-risk; **100% recommended for medical/billing** |
| **Python async pattern** | `begin_analyze_document_from_url(model_id, url)` → `task.result()` |
| **Prebuilt model call** | Same method — just pass `"prebuilt-invoice"` etc. as the model ID |
